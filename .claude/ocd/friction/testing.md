# testing

- 2026-04-24 — pytest `--import-mode=importlib` plus a regular-package `__init__.py` somewhere in the test tree (e.g., `tests/plugins/ocd/systems/__init__.py`) silently shadows the source `systems/` namespace package: any test that imports `systems.<module>._<thing>` fails with `ModuleNotFoundError` if the matching test subdirectory is missing its own `__init__.py`, because the regular-package claim on `systems` constrains submodule resolution to the test directory only, even with the source path on `pythonpath`. Expected: `pythonpath` setting in `pyproject.toml` puts source paths first and they resolve before the test-tree shadow. Workaround: when adding a new test subtree under `tests/plugins/ocd/systems/<system>/`, also add empty `__init__.py` at every directory level down to the test files so the test tree's package shape stays consistent (matches the existing convention for `check`, `governance`, etc.). Root cause: the test-tree `systems/__init__.py` makes `systems` a regular package; once regular, Python no longer searches other `pythonpath` entries for sibling submodules — see [Python docs on regular vs namespace packages](https://docs.python.org/3/reference/import.html#regular-packages).
